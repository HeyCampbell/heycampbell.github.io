Groking Procs and Blocs

Or Campbell tries to understand things that ryhme
11/14/14

Blocks are simple enough. Or so I thought. I had secretly been learning about them for a while now. I didn't know what it was called, I just saw it in some solutions and began to figure out what the wierd looking stuff inside the curly braces did. And it was so cool. suddenly I went from building a full loop with an iterator I'm doing it all in one line. Glorious.

array.each {|value_in_array| somemethod(value_in_array)}

This sort of thing was popping up left and right. Basically it allows us to go through each value in the array and apply a method to it. You can exptrapolate from there how intricate you can make these things, how many possibilities this opens up to you. But it doesn't work on every method out there. And that just made me sad that I could not build a method that worked with blocks. Then I actually read the chapters I'm supposed to read in the books I'm supposed to look at. And suddenyl it all started to blow up. I had been using blocks for the last 2 weeks without having any idea the power behind them. I knew that some methods could work with blocks and some couldn't. Now I know why.

The Yield method

It's pretty simple. This little method allows you to pass values from within an object to a block. Or whatever else it is you want to spit out to, but a block is your best bet. If that is in your method, you can use that method to send data to a block. If not, sorry buddy, no blocks for you.

Super cool, right? Wouldn't it be great if you could take that block you've just made and plug it in to a variable so you won't have to retype it over and over again? Yeah, thats what a bunch of software engineers figured when they were developing Ruby. So they gave us Procs. This allows us to wrap the block code up into a nice little piece of takeout food and use it wherever we need that block code. Procs take the place of functions in Ruby. Functors are wjhat some people call them. Unlike methods, which are more like pure actions, procs are actually objects. And this being an object oriented language, that can come in really handy.

To call a proc, you have a few options. This is Ruby after all, there are one hundred different ways to skin a method. You can call a proc similarly to a block.

proc_bar = Proc.new {|value_in_array| somemethod(value_in_array)} foo.each(%proc_bar)

Or you can simply just call it.

proc_bar.call

I know,t hats going to be really hard to remember.

Your next task, faithful reader, is to understand the difference between lambdas and procs. I'm not so great with it, but basically it boils down to this. When a lambda returns a value within a method, the program flow continues through the rest of the method, leaving the possibility of returning a different value than the lambda returns. A Proc will return the value of the proc, then return control from the method containing the proc. Or something like that. Seriously, I'm just learning this stuff here too. And I have a hard time getting knowledge to fit in my mind when I haven't found a reason to use it yet.

Got any good reasons to use lambdas? Let me know!
